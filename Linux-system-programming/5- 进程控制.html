    # 进程控制

    # 1. 进程概述

    # 1.1 程序和进程

    • 程序

      • 程序，是指编译好的二进制文件，在磁盘上，不占用系统资源(cpu、内存、打开的文件、设备、锁....)，只占用磁盘空间

    • 进程

      • 运行的程序
      • 不占用磁盘空间
      • 占用cpu和内存资源

    • 进程不占用磁盘资源

      进程，是并发执行的程序在执行过程中分配和管理资源的基本单位，是一个动态概念，竟争计算机系统资源的基本单位。每一个进程都有一个自己的地址空 间，即进程空间或（虚空间）。进程空间的大小 只与处理机的位数有关，一个 16 位长处理机的进程空间大小为 2^16^ ，而 32 位处理机的进程空间大小为 2^32^ 。

    # 1.2 并行和并发

    并发，在操作系统中，一个时间段中有多个进程都处于已启动运行到运行完毕之间的状态。但，任一个时刻点上仍只有一个进程在运行。

    例如，当下，我们使用计算机时可以边听音乐边聊天边上网。 若笼统的将他们均看做一个进程的话，为什么可以同时运行呢，因为并发。

    • 并行和并发

    并发: 在某一个比较短的时间段处理的任务的量

    • 处理的任务量越多, 并发量越高

    并行: 相当于做了集群(从一个变成了多个), 在一个时间点就可以处理多任务

    • CPU时间片

      时间片即CPU分配给各个程序的时间，每个进程被分配一个时间段，称作它的时间片，即该进程允许运行的时间，使各个程序从表面上看是同时进行的。如果在时间片结束时进程还在运行，则CPU将被剥夺并分配给另一个进程。如果进程在时间片结束前阻塞或结束，则CPU当即进行切换。而不会造成CPU资源浪费。

      • 在宏观上：我们可以同时打开多个应用程序，每个程序并行不悖，同时运行。

      • 但在微观上：由于只有一个CPU，一次只能处理程序要求的一部分，为了公平处理，方法就是引入时间片，让每个程序轮流执行。

      在Linux的内核处理过程中，每一个进程默认会有一个固定的时间片来执行命令（默认为1/100秒），这段时间内进程被分配到CPU，然后独占使用。如果使用完，同时未到时间片的规定时间，那么就主动放弃CPU的占用，如果到时间片尚未完成工作，那么CPU的使用权也会被收回，进程将会被中断挂起等待下一个时间片。

    # 1.4 进程控制块PCB

    我们知道，每个进程在内核中都有一个进程控制块（PCB）来维护进程相关的信息，Linux内核的进程控制块是task_struct结构体。

    /usr/src/linux-headers-3.16.0-30/include/linux/sched.h文件中可以查看struct task_struct 结构体定义。其内部成员有很多，我们重点掌握以下部分即可：

    • 进程id：系统中每个进程有唯一的id，在C语言中用pid_t类型表示，其实就是一个非负整数。

    • 进程的状态，有就绪、运行、挂起、停止等状态。

    • 进程切换时需要保存和恢复的一些CPU寄存器。

    • 描述虚拟地址空间的信息。

      • 保证cpu MMU进行映射

    • 描述控制终端的信息。

      • 方便输入输出

    • 当前工作目录（Current Working Directory）。

      • 程序在哪个目录下执行, 进程的工作目录就是谁

    • umask掩码。

      • 控制文件的权限
      • 在进程中重新设置掩码, 以后在进程中创建新文件就会和这个掩码进行运算

    • 文件描述符表，包含很多指向file结构体的指针。

      • 记录着当前进程中打开的文件

    • 和信号相关的信息。

    • 用户id和组id。

    • 会话（Session）和进程组。

      • 进程组: 多个进程集合, 进程组也有ID
      • 会话: 多个进程组的集合, 会话有会话ID

    • 进程可以使用的资源上限（Resource Limit）。

      • ulimit -a

    # 1.5 进程状态

    进程有五种状态:

    • 初始态 == 创建态 [从无到有]
    • 就绪态
    • 运行态
    • 挂起态 == 阻塞态
    • 终止态 == 退出 [从有到无]
    初始态 == 创建态   [从无到有]
	- 启动一个磁盘程序, 有了一个进程
就绪态
	- 进程已经被创建出来, 不能运行
	- 因为默认进程没有CPU时间片
运行态
	- 进程抢到了CPU时间片
	- 这是一个运行的进程, 开始处理业务逻辑
	- cpu时间片用完了 -> 就绪态
挂起态 == 阻塞态
	- 一般是从运行态转变过来的
	- 用户在程序中做了对应的函数调用
		- sleep(100);
	- 挂起态的进程会放弃cpu
	- 好多情况下挂起态需要手动恢复(调用对应函数)
终止态 == 退出 [从有到无]
	- 退出进程

    # 2.2 进程的创建

    • 如何在程序中创建进程?

      #include <unistd.h>
// 创建一个新的进程
// 函数调用成功之后, 进程的个数在原基础上添加一个
pid_t fork(void);
返回值:
	- 失败: -1, 子进程不会被创建
	- 成功: 子进程被创建, 有两个返回值  
    //注意返回值，不是fork函数能返回两个值，而是fork后，fork函数变为两个，父子需【各自】返回一个
		- 子进程: 返回0
		- 父进程: 返回子进程的进程ID

      1. 有一个进程, 唯一的进程 -- > A
2. 在A进程中调用了fork函数
	- 进程个数变成了两个
		- A -> parent
		- B -> child
	- 每个进程都对应一个虚拟地址空间
		- B的虚拟地址空间怎么来的?
    		- 将A的虚拟地址空间复制了一份
    - 虚拟地址空间中有什么?
    	- 内核区
    	- 用户区
    		- 代码区
    			- 结论: 父子进程各自有一份代码, 代码的内容是相同的

    • 如何区分父子进程?

      int main()
{
    pid_t pid = fork();
    // 下边开始就有两个进程, 父子进程
    // 在父子进程的代码区fork的返回值不同, 通过返回值区分父子进程
    if(pid > 0)
    {
        // 父进程
    }
    else if(pid == 0)
    {
        // 子进程
    }
}

    # 2.3 父子进程

    父子进程之间在fork后。有哪些相同，那些相异之处呢？

    • 区别

      • 父子进程的进程ID不同 -> 在内核区 pcb 中
      • 进程的状态不同 -> 内核区
        • 每个进程都需要自己去争抢cpu时间片
      • cpu的寄存器信息不同 -> 内核区

    • 共同点

      • 用户区:
        • 都一样 (在创建成功之后暂时一样)
        • 两个进程各自运行, 各自处理的逻辑不同, 各自数据也会对应发送变化
          • 两个进程之间互不影响
      • 内核区
        • 文件描述符表
          • 假设在父进程中可以通过文件描述符操作fd a.txt
          • 在子进程中也可以通过文件描述符(复制过来的)fd'操作 a.txt
          • 由此两个不相关的进程便可操作磁盘上的同一个文件，进而实现进程间通信

    • 父子进程之间如果想通信, 可不可以使用全局变量?

      • 不可用, 因为每个进程都有各自的全局变量, 虽然父子进程中的全局变量名相同，但是却没有任何关联

    • 读时共享, 写时复制

    MMU将虚拟内存映射到物理内存时数据的变化
    ## 左侧: 父进程
## 右侧: 子进程
## 父子进程用户区刚拷贝完成之后相同
## 如果父子进程只对各自数据区中的数据进行读操作:
	- 这个数据映射到物理内存时候用的是同一块地址
## 如果父子进程只对各自数据区中的数据进行写操作:
	- 这两个数据在物理内存中对应不同的存储地址
## 原则:
	读时共享, 写时复制

    • 执行多进程可执行程序, 现象：

    // 程序执行完成了, 终端的状态不太对
// 原因: 
	- ./a.out在当前终端执行, a.out对应的进程的父进程是当前终端
	- 当前终端知道有一个他的子进程 a.out
	- a.out进程执行, 创建了子进程(child), 但是当前终端不知道有child进程的存在
	- 当前终端一直在监测a.out的状态, 如果a.out进程结束了, 终端的bash切换到了前台
	- a.out结束了,但是a.out的子进程child进程没有结束, child进程如果要输出
		- 输出到当前的终端上
	- 以上的问题, 只是时间上的错乱, bash切换到前台切换早了

    # 2.4 gdb调试

    使用gdb调试的时候，gdb只能跟踪一个进程。可以在fork函数调用之前，通过指令设置gdb调试工具跟踪父进程或者是跟踪子进程。默认跟踪父进程。

    $ set follow-fork-mode child  ## 命令设置gdb在fork之后跟踪子进程
$ set follow-fork-mode parent ## 设置跟踪父进程。
## 注意，一定要在fork函数调用之前设置才有效

    # 3. execl和execlp函数

    fork创建子进程后执行的是和父进程相同的程序（但有可能执行不同的代码分支），子进程往往要调用一种exec函数以执行另一个程序。当进程调用一种exec函数时，该进程的用户空间代码和数据完全被新程序替换，从新程序的启动例程开始执行。调用exec并不创建新进程，所以调用exec前后该进程的id并未改变。

    将当前进程的.text、.data替换为所要加载的程序的.text、.data，然后让进程从新的.text第一条指令开始执行，但进程ID不变，换核不换壳。

    // 其实有六种以exec开头的函数，统称为exec族：
int execl(const char *path, const char *arg, ...);
int execlp(const char *file, const char *arg, ...);
int execle(const char *path, const char *arg, ..., char *const envp[]);
int execv(const char *path, char *const argv[]);
int execvp(const char *file, char *const argv[]);
int execve(const char *path, char *const argv[], char *const envp[]);
//事实上，只有execve是真正的系统调用，其它五个函数最终都调用execve，所以execve在man手册第2节，其它函数在man手册第3节。这些函数之间的关系如下图所示。
    exec函数族

    execl和execlp函数的作用是根据指定的文件名找到可执行文件，并用它来取代调用进程的内容，换句话说，就是在调用进程内部执行一个可执行文件。

    execl和execlp函数执行成功后不会返回，因为调用进程的实体，包括代码段，数据段和堆栈等都已经被新的内容取代，只留下进程ID等一些表面上的信息仍保持原样，颇有些神似"三十六计"中的"金蝉脱壳"。看上去还是旧的躯壳，却已经注入了新的灵魂。只有调用失败了，它们才会返回一个-1，从原程序的调用点接着往下执行。

    // 通过execl或者execlp 启动磁盘上的可执行程序 -> 得到一个新的进程 -> 进程的总个数还是1个
// 工作原理:
	- 现在有一个进程 A 
	- 在进程A中调用了execl或者execlp , 启动了磁盘上的可执行程序 `ps aux`
	- 函数调用成功之后, A进程的虚拟地址空间数据发生了变化
		- A进程的代码段的代码全部被替换了 -> 变成了ps程序中的代码段的代码
			- 栈区数据发生变化
			- 堆区发生变化
			- 全局数据区数据发生变化
			- .....
        - 最终导致A虚拟地址空间用户区的数据基本上全变了
        
// 如何使用?
- 进程A在运行, 如果想用启动一个磁盘程序 ps
- 创建A进程的子进程B
- 让A继续执行自己的业务逻辑
- 在子B进程中 调用 execlp / execl
	- 虽然B进程的代码段别替换了, 但是A和B仍然是父子关系

    // 函数原型
#include <unistd.h>
extern char **environ;
// l --> list
// 一般使用execl启动磁盘上的自己编译出的可执行程序
// 启动系统自带的可执行程序也是可以的, 但是必须要指定路径
int execl(const char *path, const char *arg, .../* (char  *) NULL */);
	参数:
		- path: 要启动的磁盘上的可执行程序(必须带路径)
            - 绝对路径: 建议
            - 相对路径: 程序如果改变存储位置, path有可能就找不到了(不建议)
        - arg: 这是一个字符串描述(启动的进程名字), 出现在 ps aux 的 COMMOND 列
        - ...: 启动的可执行程序的参数(>=0), 如果参数结束了, 末尾使用NULL(哨兵)
            如果有多个参数: 
					"a", "u", "x", NULL
// p --> path
// 启动系统自带的可执行程序 -> 不需要指定路径, 直接写文件名即可
// 启动磁盘上的自己编译出的可执行程序也是可以的 -> 指定路径
int execlp(const char *file, const char *arg, .../* (char  *) NULL */);
	参数: 
		- file: 要启动的磁盘上的可执行程序的名字(只写名字就可以)
            - 指定的这个名字在执行的时候, 会搜索环境变量 -> PATH (对应的值是n个路径)
            	- 搜索这些路径中有没有叫file的可执行程序
        - arg: 这是一个字符串描述(启动的进程名字), 出现下 ps aux 的 COMMOND 列
        - ...: 启动的可执行程序的参数(>=0), 如果参数结束了, 末尾使用NULL(哨兵)
            如果有多个参数: 
					"a", "u", "x", NULL
// 以上两个函数共同的特点:
	- 函数执行成功之后不会返回
	- 函数执行失败了返回 -1
// 所以通常我们直接在exec函数调用后直接调用perror()和exit()，无需if判断。

    示例

    #include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <string.h>

int main()
{
    //1、创建一个子进程
    pid_t pid = fork();
    if(pid > 0) {
        printf("parent pid:%d\n",getpid());
    }else if(pid == 0) {
        printf("child pid:%d\n",getpid());
        
        //2、调用磁盘程序
        //int ret = execl("/bin/ps","ps-test","aux",NULL);
        int ret = execlp("ps","ps-test","a","u","x",NULL);
        printf("ret = %d\n",ret);
        perror("execl");
    }

    return 0;
}

    # 4. 进程控制

    父进程通过fork创建了子进程

    • 虚拟地址空间被复制
      • 用户区
      • 内核区
        • pcb (进程控制块)

    子进程在销毁(自动退出)的时候

    • 将用户区资源释放
    • 内核区的pcb子进程释放不了, 需要交给父进程去处理

    # 4.2 孤儿进程

    父进程创建子进程

    • 父进程英年早逝
    • 子进程早年丧父 -> 没有爹了
    • 子进程死了之后, 需要有进程回收它的pcb资源
      • 系统的Init进程会领养这些孤儿
        • init进程在linux中是 1号 进程
        • 有些操作系统并不是init进程对孤儿进程进行"领养"，而是交给桌面进程"领养"

    # 4.3 僵尸进程

    僵尸进程不是一个活的进程, 如何得到一个僵尸进程?

    • 父进程创建子进程
    • 子进程全部都死了
    • 父进程还活着, 父进程理论上需要回收子进程资源
      • 父进程不干活, 就是不回收子进程的pcb
      • 这时候子进程就变成了僵尸(Zombie)进程

    # 4.4 进程回收

    一个进程在终止时会关闭所有文件描述符，释放在用户空间分配的内存，但它的PCB还保留着，内核在其中保存了一些信息：如果是正常终止则保存着退出状态，如果是异常终止则保存着导致该进程终止的信号是哪个。这个进程的父进程可以调用wait或waitpid获取这些信息，然后彻底清除掉这个进程。我们知道一个进程的退出状态可以在Shell中用特殊变量$?查看，因为Shell是它的父进程，当它终止时Shell调用wait或waitpid得到它的退出状态同时彻底清除掉这个进程。

    • wait

      #include <sys/types.h>
#include <sys/wait.h>
//父进程调用wait函数可以回收子进程终止信息。该函数有三个功能：
//① 阻塞等待子进程退出 
//② 回收子进程残留资源 
//③ 获取子进程结束状态(退出原因)。
pid_t wait(int *status);
    功能：
        等待任意一个子进程结束，如果任意一个子进程结束了，此函数会回收该子进程的资源。
    参数：
        status : 传出参数,返回进程退出时的状态信息。
    返回值：
        成功：已经结束子进程的进程号
        失败： -1

      示例

      #include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <string.h>
#include <sys/wait.h>

int main()
{
    pid_t pid = 0;
    // 循环创建9个子进程
    for(int i=0; i<9;i++){
        pid = fork();
        if(pid == 0){
            break;
        }
    }

    if(pid > 0){
        //父进程
        while(1)
        {
            pid_t ret = wait(NULL);
            if(ret > 0){
                printf("child die,pid:%d\n",ret);
            }
            else if(ret == -1){
                // 失败或者没有子进程
                printf("失败或者没有子进程\n");
                break;
            }
        }
    }
    else{
        // 子进程
        printf("子进程的PID:%d,父进程的PID:%d\n",getpid(),getppid());
    }

    return 0;
}

    • waitpid

      #include <sys/types.h>
#include <sys/wait.h>

// 该函数可以设置为阻塞或非阻塞
// 可以指定回收哪一个子进程的资源,也可以直接回收所有进程资源
// 不管阻塞还是非阻塞, 没有子进程返回-1
pid_t waitpid(pid_t pid, int *status, int options);
	参数:
		- pid: 
			 >0: 某个进程的进程ID     ->>>>>  经常用的
             -1: 回收任意子进程（相当于wait）
			<-1: 取绝对值, 作为某个进程组的组ID, 回收的是这个组中的所有子进程资源
			  0: 回收当前进程所在的进程组的子进程资源
		- status: 传出参数, 返回进程退出时候的一些状态信息, 一般时候用不着, 传NULL
		- options: 控制当前函数阻塞或者非阻塞
			- WNOHANG: 非阻塞
			- 0: 阻塞的
	返回值:
		失败: -1,没有子进程
        成功: 
			>0: 回收的子进程的进程ID
      		=0: 参3为WNOHANG，且子进程正在运行

      示例

      #include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <string.h>
#include <sys/wait.h>

int main()
{
    pid_t pid = 0;
    // 循环创建9个子进程
    for(int i=0; i<9;i++)
    {
        pid = fork();
        if(pid == 0){
            break;
        }
    }

    if(pid > 0){
        //父进程
        while(1)
        {
            // 阻塞
            //pid_t ret = waitpid(-1,NULL,0);
            // 非阻塞
            pid_t ret = waitpid(-1,NULL,WNOHANG);
            if(ret > 0)
            {
                printf("child die,pid:%d\n",ret);
            }
            else if(ret == 0)
            {
                printf("子进程还活着...\n");  //空轮询
            }
            else if(ret == -1)
            {
                // 失败或者没有子进程
                printf("失败或者没有子进程\n");
                break;
            }
        }
    }
    else{
        // 子进程
        printf("子进程的PID:%d,父进程的PID:%d\n",getpid(),getppid());
        sleep(1);
    }

    return 0;
}

    • wait / waitpid 的第二个参数

      // wait / waitpid 的第二个参数status
pid_t wait(int *status);
pid_t waitpid(pid_t pid, int *status, int options);
// 可使用wait函数传出参数status来保存进程的退出状态。借助宏函数来进一步判断进程终止的具体原因。

/*
    宏函数可分为如下三组：
     1. WIFEXITED(status) 为非0	→ 进程正常结束
        WEXITSTATUS(status) 如上宏为真，使用此宏 → 获取进程退出状态 (exit的参数)
     2. WIFSIGNALED(status) 为非0 → 进程异常终止
        WTERMSIG(status) 如上宏为真，使用此宏 → 取得使进程终止的那个信号的编号。
     3. WIFSTOPPED(status) 为非0 → 进程处于暂停状态
        WSTOPSIG(status) 如上宏为真，使用此宏 → 取得使进程暂停的那个信号的编号。
        WIFCONTINUED(status) 为真 → 进程暂停后已经继续运行
*/

参数: status
	- 判断进程是不是正常退出, 并得到退出时候的状态码
		main函数中的return或者exit函数
			- WIFEXITED(status): 是不是正常退出
		得到退出时候的状态码:
			- WEXITSTATUS(status)
                
	- 判断进程是不是被信号杀死了, 并且得知进程是被哪个信号杀死的
		- 是不是被信号杀死了:
			- WIFSIGNALED(status)
        - 得到信息是被哪个信号杀死了:
			- WTERMSIG(status)

      示例

      #include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <string.h>
#include <sys/wait.h>

int main()
{
    pid_t pid = 0;
    // 循环创建9个子进程
    for(int i=0; i<9;i++){
        pid = fork();
        if(pid == 0){
            break;
        }
    }

    if(pid > 0){
        //父进程
        while(1){
            int status;
            // 阻塞
            //pid_t ret = waitpid(-1,NULL,0);
            // 非阻塞
            pid_t ret = waitpid(-1,&status,WNOHANG);
            if(ret > 0)
            {
                printf("child die,pid:%d\n",ret);
                // 判断子进程是不是正常死亡
                if(WIFEXITED(status))
                {
                    printf("进程退出的状态码:%d\n",WEXITSTATUS(status));
                }
                else if(WIFSIGNALED(status))
                {
                    printf("被%d信号杀死了...\n",WTERMSIG(status));
                }
            }
            else if(ret == 0)
            {
                //printf("子进程还活着...\n");
            }
            else if(ret == -1)
            {
                // 失败或者没有子进程
                printf("失败或者没有子进程\n");
                break;
            }
        }
    }else{
        // 子进程
        while(1){
            printf("子进程的PID:%d,父进程的PID:%d\n",getpid(),getppid());
            sleep(9);
        }
    }

    return 99;
}

{

        if(pid == 0)

            break;
        }

                }
            }
            else if(ret == -1)

                break;
            }
        }

    else if(i == 0)
    {
        // 子进程
