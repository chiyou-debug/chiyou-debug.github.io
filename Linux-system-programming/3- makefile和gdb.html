              # Makefile && GDB

              # 1. Makefile

              什么是makefile？或许很多Winodws的程序员都不知道这个东西，因为那些Windows的IDE都为你做了这个工作，但我觉得要作一个好的和professional的程序员，makefile还是要懂。这就好像现在有这么多的HTML的编辑器，但如果你想成为一个专业人士，你还是要了解HTML的标识的含义。特别在Unix下的软件编译，你就不能不自己写makefile了，会不会写makefile，从一个侧面说明了一个人是否具备完成大型工程的能力。

              因为，makefile关系到了整个工程的编译规则。一个工程中的源文件不计数，其按类型、功能、模块分别放在若干个目录中，makefile定义了一系列的规则来指定，哪些文件需要先编译，哪些文件需要后编译，哪些文件需要重新编译，甚至于进行更复杂的功能操作，因为makefile就像一个Shell脚本一样，其中也可以执行操作系统的命令。

              makefile带来的好处就是——“自动化编译”，一旦写好，只需要一个make命令，整个工程完全自动编译，极大的提高了软件开发的效率。make是一个命令工具，是一个解释makefile中指令的命令工具，一般来说，大多数的IDE都有这个命令，比如：Visual C++的nmake，Linux下GNU的make。

              # 1.1 文件命名

              • 两种书写格式:
                • makefile
                • Makefile

              # 1.2 makefile中的规则

              ## 在一个makefile文件中, 可以有多条规则, 规则中规定了代码如何进行构建, 
## 生成的是库还是可执行程序或者是其他东西
## 规则的语法
目标 ...:依赖 ...
	命令 (必须要有一个tab缩进)
	...
## 目标: 可以有多个, 最终要生成的文件 (这个目标也有可能不会生成)
## 依赖: 生成目标的原材料
## 命令: 通过一个指令将依赖生成目标

## 举例:
gcc a.c b.c c.c d.c -o app
## 将上述操作写到makefile中
app:a.c b.c c.c d.c
	gcc a.c b.c c.c d.c -o app
	
## 如果makefile文件中有多条规则, 我们只想执行某条规则中的命令
 	make 这条规则中的目标即可

              # 1.3 makefile中的知识点

              # 1.3.1 工作原理

              • 关于依赖检测
                • 在执行makefile中的命令的时候, 会首先检测命令使用的依赖是否存在
                • 若不存在，make会向下检测其他的规则, 看一看有没有其他规则是生成这个依赖的
                  • 这条新的规则中的目标是其他规则中的依赖
                • 如果有就执行这条规则中的命令生成对应的目标 -> 需要的依赖就有了
                • 如果makefile中有多条规则, 执行make命令时, 会自动查找makefile中第一条规则, 执行规则中的命令
              • 关于文件更新的检测
                • 更新的是依赖文件或者是目标文件
                • 比较两个文件的日期
                  • 依赖文件 和 目标文件
                    • 目标文件日期 比 依赖文件日期 要晚 ==> 正常状态
                    • 目标文件日期 比 依赖文件日期 要早 ==> 不正常
                      • 重新生成目标, 执行对应的规则中的命令即可
                  • 如果目标不存在，执行规则中对应的命令来生成目标

              # 1.3.2 变量

              • 自定义变量

                ## makefile中自定义变量, 一般小写
变量名=变量值
## 如何取变量的值
$(变量名)
## 举例
var=1234
var1=$(var)

              • 默认的自带变量

                ## makefile中自带的变量一般都是大写的
## 举例
CC ## == gcc
$(CC)

              • 自动变量

                ## 自动变量只能在规则中的命令里边使用
$@: 规则中的目标
$<: 规则中第一条依赖
$^: 规则中的所有依赖

## 例子
app:add.o div.o main.o mult.o  sub.o
        gcc add.o div.o main.o mult.o  sub.o -o app
## 修改
app:add.o div.o main.o mult.o  sub.o
        gcc $^ -o $@

              # 1.3.3 模式匹配

              add.o:add.c
        gcc -c add.c
sub.o:sub.c
        gcc -c sub.c
main.o:main.c
        gcc -c main.c
mult.o:mult.c
        gcc -c mult.c
div.o:div.c
        gcc -c div.c
        
## 规律, % 匹配文件名, 前后两个%对应同一个文件名
%.o:%.c
	gcc -c $< -o $@

              # 1.3.4 函数

              makefile中所有的函数都是有返回值的, 因此所有函数的样式:

              $(函数名 参数1, 参数2, 参数3, ...)

              • 获取指定目录下指定类型的文件列表 -- wildcard

                $(wildcard PATTERN...) 
PATTERN: 指定文件所在的路径,路径可以有一个或多个,如果有多个路径使用空格间隔
		 文件的类型是通过文件的后缀判定的
		 路径的格式: /home/chiyou/a/*.c  /home/chiyou/b/*.c  ./*.c
返回值: n个满足条件的文件名的列表
	/home/chiyou/a/a.c /home/chiyou/a/b.c /home/chiyou/b/c.c ./aa.c ./bb.c
                • 功能:
                  • PATTERN 指的是某个或多个目录下的对应的某种类型的文件
                  • 如果有多个目录，一般使用空格间隔
                • 返回值：
                  • 得到的若干个文件的文件列表， 文件名之间使用空格间隔
                • 示例：$(wildcard *.c ./sub/*.c)
                  • 返回值格式: a.c b.c c.c d.c e.c f.c ./sub/aa.c ./sub/bb.c

              • 模式字符串替换函数——patsubst

                $(patsubst <pattern>,<replacement>,<text>)
参数有三个:
	- pattern: 要将哪一部分进行替换
	- replacement: 要将第一个参数的字符串替换成什么(新的字符串)
	- text: 原材料, 要被替换的数据
## 举例:
src=/home/chiyou/a/a.c /home/chiyou/a/b.c /home/chiyou/b/c.c ./aa.c ./bb.c  
## /home/chiyou/a/a.o /home/chiyou/a/b.o /home/chiyou/b/c.o ./aa.o ./bb.o
obj=$(patsubst %.c,%.o,$(src))
                • 功能:
                  • 查找<text>中的单词(单词以“空格”、“Tab”或“回车”“换行”分隔)是否符合模式<pattern>，如果匹配的话，则以<replacement>替换。
                  • 这里，<pattern>可以包括通配符%，表示任意长度的字串。如果<replacement>中也包含%，那么，<replacement>中的这个 %将是<pattern>中的那个%所代表的字串。(可以用“\”来转义，以\%来表示真实含义的%字符)
                • 返回值:
                  • 函数返回被替换过后的字符串。
                • 示例: $(patsubst %.c, %.o, x.c.c bar.c)
                  • 把字串“x.c.c bar.c”符合模式[%.c]的单词替换成[%.o]
                  • 返回结果是“x.c.o bar.o”

              # 2. GDB调试

              gdb 是由 GNU 软件系统社区提供的调试器，同 gcc 配套组成了一套完整的开发环境，可移植性很好，支持非常多的体系结构并被移植到各种系统中（包括各种类 Unix 系统与 Windows 系统里的 MinGW 和 Cygwin ）。此外，除了 C 语言之外，gcc/gdb 还支持包括 C++、Objective-C、Ada 和 Pascal 等各种语言后端的编译和调试。 gcc/gdb 是 Linux 和许多类 Unix 系统中的标准开发环境，Linux 内核也是专门针对 gcc 进行编码的。

              gdb 的吉祥物是专门捕杀 bug 的射手鱼:

              For a fish, the archer fish is known to shoot down bugs from low hanging plants by spitting water at them.

              # 2.1 调试准备

              通常，在为调试而编译时，我们会(在尽量不影响程序行为的情况下)关掉编译器的优化选项(-O)， 并打开调试选项(-g)。另外，-Wall选项打开所有 warning，也可以发现许多问题，避免一些不必要的 bug:

              ## -g参数在生成的可执行程序中加入调试信息
$ gcc -g -Wall program.c -o program

              -g选项的作用是在可执行文件中加入源代码的信息，比如可执行文件中第几条机器指令对应源代码的第几行，但并不是把整个源文件嵌入到可执行文件中，所以在调试时必须保证gdb能找到源文件。
